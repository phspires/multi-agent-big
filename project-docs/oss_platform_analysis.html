Enquadramento

Porque AWARE e mindLAMP não são equivalentes — e porque isso importa

A confusão entre AWARE e mindLAMP é compreensível — ambos recolhem dados passivos de smartphone, ambos são open-source, e ambos foram usados em estudos de fenotipagem mental. Mas servem propósitos fundamentalmente diferentes e estão em camadas distintas da arquitectura de um produto.

A distinção central: AWARE é uma framework de recolha de dados brutos (sensing layer), enquanto mindLAMP é uma plataforma de investigação e cuidado clínico completa que inclui sensing, EMA, entrega de intervenção, e dashboard clínico. Usar ambos em simultâneo seria redundante e criaria conflito de dados. A escolha entre eles — ou entre qualquer das alternativas — depende do que se quer construir.

O campo tem pelo menos seis plataformas open-source relevantes, cada uma com uma filosofia técnica e um caso de uso óptimo distintos: AWARE, mindLAMP, Beiwe, RADAR-base, m-Path Sense, e a combinação Medplum+custom. A análise que se segue compara-as rigorosamente em todas as dimensões relevantes para a decisão de arquitectura de produto.

Framework Analítica

As três camadas funcionais de um produto PerSense

Antes de comparar plataformas, é essencial entender que um produto de fenotipagem digital + JITAI precisa de cobrir três camadas funcionais distintas. Nenhuma plataforma cobre as três igualmente bem — a decisão de arquitectura é escolher que combinação minimiza complexidade e dependências.

Camada 1 — Sensing & Data Collection
Passive smartphone sensors (GPS, calls, BT, screen) Wearable data (Fitbit/Garmin/Apple Watch) Active EMA (ecological momentary assessment) Clinical questionnaires (PHQ-9, GAD-7, ISI…) GDPR-compliant storage (EU server)
↓
Camada 2 — ML / Analytics / JITAI Engine
Feature engineering (biweekly windows, circadian) Binary/continuous prediction models (PHQ-9/GAD-7) Phenotype discovery (clustering) JITAI decision rules (timing + content triggers) Model monitoring + drift detection
↓
Camada 3 — Intervention Delivery & Clinical Interface
CBT/ACT module delivery (iNNOVBC content) Adaptive module selection (JITAI output) Therapist dashboard (progress, alerts, comms) Patient-facing app (progress visualisation) Secure therapist-patient messaging Clinical data for MDR documentation

Os componentes em destaque (JITAI engine + adaptive module selection) são o IP diferenciador que nenhuma plataforma open-source fornece — têm de ser construídos de raiz independentemente do stack escolhido.

Papel na arquitectura PerSense

AWARE é apenas Camada 1. É a melhor escolha para passive sensing de smartphone (especialmente Android) e a CMS já tem experiência operacional com ela. Não deve ser combinada com mindLAMP — criariam conflito de dados e duplicação de sensores. Escolher AWARE significa construir Camada 2 (ML) e Camada 3 (intervenção) separadamente.

Recomendação de uso

Manter AWARE para Camada 1 se se construir arquitectura modular. A reutilização da app e do pipeline do iNNOVSensing poupa 4–6 semanas de desenvolvimento. Custo zero em licenças. Mas requer solução separada para entrega de intervenção e dashboard clínico.

Papel na arquitectura PerSense

mindLAMP cobre Camadas 1 e 3 de forma integrada, com suporte parcial à Camada 2 via Cortex. É a única plataforma que combina sensing + EMA + entrega de intervenção + dashboard clínico numa solução coesa. O facto de John Torous ser simultaneamente co-PI do PRODROMUS e o maintainer do projecto é uma vantagem estratégica única — acesso directo a suporte técnico e potencial de co-desenvolvimento.

Recomendação de uso

mindLAMP é a escolha central para PerSense/PRODROMUS. Substitui iTerapi para Camada 1+3, é o backbone do PRODROMUS por decisão do co-PI, e tem Cortex para analytics. A única razão para não usar mindLAMP seria se se precisar de raw sensor data para features muito granulares — nesse caso, complementar com AWARE apenas para sensing bruto, mas manter mindLAMP para intervenção e dashboard.

Papel na arquitectura PerSense

Beiwe é óptimo para investigação que precisa de raw sensor data de alta qualidade e reproducibilidade científica máxima. Para o PerSense como produto, tem dois problemas: (1) não entrega intervenções, (2) infraestrutura AWS complexa e cara. Recomendação de uso

Não usar como plataforma principal. A complexity de deployment AWS + ausência de funcionalidades clínicas não justifica vs mindLAMP. Considerar usar a Forest library (open-source, Python) para feature engineering mesmo que não se use o resto da plataforma.

Papel na arquitectura PerSense

RADAR-base é a plataforma mais alinhada com os requisitos de um ensaio clínico europeu de larga escala (exactamente o que o PerSense FCT propõe). O suporte nativo a JITAI triggering é um diferenciador real — nenhuma outra plataforma tem isto out-of-the-box. A sua fraqueza é a complexidade de deployment: não é adequada para um MVP, mas pode ser a escolha certa para a fase de RCT quando se tiver um CTO e infraestrutura estabilizada.

Recomendação de uso

Considerar para Fase 2 (RCT PerSense/FCT), não para MVP. A complexidade técnica é demasiada para a fase inicial. Quando o projecto tiver um CTO dedicado e o RCT estiver a ser planeado, RADAR-base é a escolha mais sólida para um ensaio clínico multicentro europeu com JITAIs reais. Contactar KCL/Maudsley BRC para colaboração técnica — há precedentes de colaborações académicas sobre RADAR-base.

Papel na arquitectura PerSense

Medplum não é uma alternativa às outras plataformas — é uma camada complementar para a Camada 3 (entrega de intervenção e clinical backend). A vantagem principal: FHIR-native facilita enormemente a documentação técnica MDR e a eventual integração com EHRs hospitalares. Se o objectivo é construir algo que passe num processo de certificação MDR, ter os dados de outcomes clínicos em FHIR desde o início é um investimento que se paga.

Recomendação de uso

Usar como clinical backend (Camada 3) em combinação com mindLAMP (Camadas 1+2). mindLAMP para sensing + EMA + analytics, Medplum para gestão de pacientes, outcomes clínicos, e entrega de módulos CBT. Esta combinação é tecnicamente mais complexa mas regulatoriamente mais robusta para MDR.

Decisão de Arquitectura

Três stacks recomendadas — por fase e contexto

Stack A — MVP rápido (0–6 meses) — Para validar o produto antes de regulação

Camada 1 — Sensing
mindLAMP app GPS, calls, screen, BT, steps Fitbit via API EMA diária (PHQ-2, GAD-2)
↓
Camada 2 — ML + JITAI
Cortex (mindLAMP analytics) scikit-learn + XGBoost JITAI engine custom (Python) Regras baseadas em PHQ-9/GAD-7 thresholds
↓
Camada 3 — Intervenção + Clínico
mindLAMP LAMP tabs (Learn/Assess/Manage/Prevent) iNNOVBC modules como LAMP activities mindLAMP clinician dashboard Mensagens via mindLAMP
↓
Infraestrutura
Hetzner Cloud (Alemanha, GDPR) Docker Compose mindLAMP self-hosted €30–50/mês infra

Vantagens: Uma única plataforma (mindLAMP) para tudo. Torous é co-PI e maintainer → suporte directo. PRODROMUS já usa este stack → código reutilizável. Deployment em <1 semana por um developer com experiência em Docker. O IP próprio é apenas o JITAI engine e os módulos iNNOVBC adaptados.

Limitações: mindLAMP não dá raw sensor data para todos os features do iNNOVSensing (ex: typing patterns detalhados). Se esses features forem críticos para o modelo, esta stack precisa de ser complementada.

Stack B — Fidelidade máxima ao iNNOVSensing (6–18 meses) — Para continuar exactamente o que foi construído

Camada 1 — Sensing (raw data)
AWARE Framework (rebuild da app iNNOVSensing) GPS, calls, messages, BT, screen — raw Fitbit API AWARE Server (self-hosted, Hetzner)
↓
Camada 2 — ML pipeline (re-implementado)
Pipeline Python re-escrito (método iNNOVSensing) Feature engineering biweekly (AWARE raw → features) scikit-learn + XGBoost (binary + regression) JITAI engine custom Beiwe Forest library (feature templates)
↓
Camada 3 — Intervenção + EMA (separada do sensing)
Medplum (FHIR backend) iNNOVBC como FHIR CarePlan + custom PWA EMA via Medplum Questionnaire resource Dashboard clínico custom (React)

Vantagens: Continuidade exacta com iNNOVSensing (mesmos sensores, mesmas features, modelos comparáveis). Raw sensor data permite features mais granulares e reprodutibilidade científica. Medplum + FHIR é a escolha mais defensável para MDR.

Limitações: Mais componentes = mais complexidade de desenvolvimento e manutenção. Requer developer com experiência em AWARE + Medplum + Python ML. Não há uma única plataforma que unifique tudo — gestão de dois stacks separados (sensing vs intervenção).

Stack C — Fase RCT / MDR (18–36 meses) — Para ensaio clínico multicentro e certificação

Stack completa para PerSense RCT + MDR
RADAR-base (sensing + JITAI triggering nativo + GDPR) mindLAMP integrado para content delivery Medplum para gestão clínica e FHIR documentation Pipeline ML custom (scikit-learn) JITAI engine integrado com Kafka Streams

Vantagens: RADAR-base foi construída para ensaios clínicos europeus de larga escala — exactamente o que o PerSense FCT propõe. JITAI triggering nativo via Kafka. Máxima GDPR compliance. Precedentes de uso em estudos IMI/EU com supervisão regulatória.

Limitações: Alta complexidade técnica (Kafka + Kubernetes). Só faz sentido quando há um CTO dedicado e o ensaio está planeado em detalhe.

Recomendação Final

Começar com Stack A (mindLAMP only) para o MVP — é a mais rápida, mais simples, e tem a ligação estratégica mais forte (Torous). Quando o PerSense FCT estiver aprovado e o CTO contratado, migrar para Stack C (RADAR-base) para o RCT formal. A Stack B é um caminho intermédio válido se houver razão técnica específica para precisar de raw sensor data (ex: querer replicar exactamente o pipeline do iNNOVSensing antes de re-escrever o modelo ML).

Síntese

Resposta directa à questão AWARE vs mindLAMP

São complementares, não concorrentes — mas para um produto comercial, mindLAMP substitui AWARE por ser mais completo (sensing + EMA + intervenção + dashboard num único sistema). AWARE é superior em raw sensor data, o que pode ser relevante para features muito granulares, mas para o perfil de features do iNNOVSensing (location entropy, call patterns, sleep, HR) mindLAMP é suficiente.

O argumento decisivo para mindLAMP como plataforma principal não é técnico — é estratégico: John Torous é simultaneamente co-PI do PRODROMUS da CMS e o maintainer do projecto mindLAMP. Isto é uma relação de suporte técnico e co-desenvolvimento que nenhuma outra plataforma pode oferecer, e que tem valor económico real (evita contratar um backend developer para manter infraestrutura customizada nos primeiros 12 meses).

O JITAI engine — a componente que adapta a intervenção em tempo real com base nos modelos ML — é o único elemento que nenhuma plataforma fornece de forma completa e que tem de ser construído como IP próprio. É aqui que o verdadeiro trabalho de desenvolvimento da startup está, e é aqui que reside a diferenciação competitiva real.
